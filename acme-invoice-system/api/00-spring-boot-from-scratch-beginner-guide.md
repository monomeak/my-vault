# Spring Boot from scratch — beginner guide

## What is Spring Boot?

Spring is a giant Java toolbox — database access, security, web servers, all of it. On its own it needs a lot of manual setup.

Spring Boot sits on top and does that setup for you. Two ideas make this work:

- **Starters** — one dependency that pulls in a whole bundle of related libraries. Add `spring-boot-starter-web`, and you get a web server, JSON support, and routing, all at once, already working together.
- **Autoconfiguration** — Spring Boot looks at what starters you added and configures sensible defaults automatically. You didn't write "start a web server on port 8080" anywhere — it's implied by adding the web starter.

That's really the whole trick: pick starters, get working defaults, override only what you need to change.

## What is Spring Initializr?

[start.spring.io](https://start.spring.io) generates a new Spring Boot project for you — think of it like `create-react-app`, but for Java. You tell it your project name and which starters you want, and it hands you a working folder: build file, starter class, correct folder layout.

This project was generated by calling that same service directly:

```bash
curl -G "https://start.spring.io/starter.zip" \
  --data-urlencode "groupId=com.acme.invoice" \
  --data-urlencode "artifactId=api" \
  --data-urlencode "packageName=com.acme.invoice" \
  --data-urlencode "javaVersion=21" \
  --data-urlencode "dependencies=web,data-jpa,security,validation,postgresql,flyway,actuator,springdoc-openapi,testcontainers" \
  -o api.zip
```

- `groupId` — your company/namespace, e.g. `com.acme.invoice`
- `artifactId` — this project's own name, e.g. `api`
- `packageName` — the Java package everything lives in
- `dependencies` — the list of starters you want

## `pom.xml` — the shopping list

`pom.xml` tells Maven (the build tool) two things: what libraries this project needs, and how to build it. Think of it as a shopping list plus assembly instructions.

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>4.0.7</version>
</parent>
```
This line is why you don't have to pick a version number for every single dependency below — the parent already knows which versions of everything work well together.

```xml
<dependencies>
    ...
</dependencies>
```
The actual list of starters — see below for what each one does.

## Every dependency, explained simply

| Dependency | In plain words |
|---|---|
| **web (webmvc)** | Gives you a working web server. Without it, there's no way to receive HTTP requests at all. |
| **data-jpa** | Lets Java classes represent database rows, and gives you ready-made methods like `save()`, `findById()` — no SQL required for basic stuff. |
| **flyway** | Manages your database's tables through version-controlled `.sql` files. Covered in depth in [[05-flyway-deep-dive]]. |
| **security** | Locks your app down by default — nothing is public until you say so. |
| **validation** | Lets you say "this field can't be blank" or "this must be a valid email" with one annotation, instead of writing that check by hand every time. |
| **postgresql** | The actual translator that lets Java talk to a Postgres database. |
| **actuator** | Adds a `/health` endpoint (and others) so you — or Docker — can check "is this app actually alive and working?" |
| **springdoc-openapi** | Reads your API code and auto-generates documentation for it — a page other developers (or the frontend team) can open to see every endpoint. |
| **testcontainers** | Lets tests spin up a real, temporary Postgres in Docker just for that test run — more realistic than fake/mocked databases. |

## `mvnw` — you don't need Maven installed

`./mvnw` is a script sitting in the project folder. The first time you run it, it downloads the exact right version of Maven for you and uses that from then on. That means:

- You never have to install Maven yourself
- Everyone on the team builds with the identical version — no "works on my machine" surprises

Always run `./mvnw ...`, never a plain `mvn ...`.

## Folder structure

```
apps/api/
├── mvnw                  — the wrapper script
├── pom.xml               — the shopping list
├── src/main/java/...     — your actual code
├── src/main/resources/
│   ├── application.properties   — settings
│   └── db/migration/            — Flyway's .sql files
└── src/test/java/...     — tests
```

This layout (`src/main/java`, `src/main/resources`, `src/test/java`) is a Maven convention — every Maven project looks like this, so once you know it here, you know it everywhere.

## What happens when you type `./mvnw spring-boot:run`

1. Your Java code gets compiled.
2. Maven downloads any dependencies it doesn't already have cached.
3. Your app's `main()` method runs.
4. Spring Boot scans everything on the classpath and wires it all together (autoconfiguration).
5. Flyway checks your database and runs any new migration files.
6. If nothing failed, you see `Started ApiApplication in X seconds` — and the app is now listening for real requests.

`application.properties` is how you tweak step 4 without writing Java — things like which port to run on, or which database to connect to.
